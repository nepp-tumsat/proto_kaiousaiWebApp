# 📱 FEチーム（フロントエンド）向けガイド

## FEチームとは？

フロントエンド（FE）は、ユーザーが実際に見て操作する画面部分を作るチームです。スマホアプリの「見た目」と「操作感」を担当します。

## 必要な知識・スキル

### 必須スキル

- **React基礎**: コンポーネント、useState、useEffectの理解
- **TypeScript基礎**: 型の基本（string, number, booleanなど）
- **JavaScript基礎**: 配列操作（map, filter）、非同期処理（async/await）

### 学習リソース（初心者向け）

- [React公式チュートリアル](https://ja.react.dev/learn) - 基礎から学べる
- [TypeScriptハンドブック](https://www.typescriptlang.org/ja/docs/handbook/intro.html) - 型の基本
- [MDN Web Docs - fetch API](https://developer.mozilla.org/ja/docs/Web/API/Fetch_API) - API呼び出しの基礎

## 用語解説

- **API（Application Programming Interface）**: サーバーからデータを取得するための「窓口」。レストランの注文窓口のようなもの
- **fetch**: JavaScriptでAPIを呼び出すための関数
- **非同期処理**: データを取得する間、他の処理を止めない仕組み
- **ローディング状態**: データを取得している間、ユーザーに「読み込み中」と表示すること
- **エラーハンドリング**: ネットワークエラーなどが起きた時の対処方法

### タスク1: APIクライアント実装

**何をするか**: 実際にAPIを呼び出すコードを書く**ステップ1: 基本のfetch関数を作る**`src/api/client.ts` を作成:

```typescript
// APIのベースURL（開発環境用）
const API_BASE_URL = 'http://localhost:3000/api'

// エラーハンドリング付きのfetch関数
export async function apiRequest<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  try {
    const response = await fetch(`${API_BASE_URL}${endpoint}`, options)
    
    // エラーチェック: ステータスコードが200番台でない場合
    if (!response.ok) {
      throw new Error(`API Error: ${response.status}`)
    }
    
    // JSONデータを取得
    const data = await response.json()
    return data
  } catch (error) {
    // エラーを再スロー（呼び出し元で処理）
    throw error
  }
}
```

**ステップ2: 店舗API関数を作る**`src/api/shops.ts` を作成:

### タスク2: Map.tsxをAPI連携に変更

**現在のコード**（`src/components/Map.tsx`）:

```typescript
import shopsData from '../data/shops.json'  // ← これを削除

function Map() {
  const [shops] = useState<Shop[]>(shopsData as Shop[])  // ← これを変更
  // ...
}
```

**変更後のコード**:

```typescript
import { getShops } from '../api/shops'  // ← API関数をインポート

function Map() {
  const [shops, setShops] = useState<Shop[]>([])
  const [loading, setLoading] = useState(true)  // ローディング状態
  const [error, setError] = useState<string | null>(null)  // エラー状態

  useEffect(() => {
    // コンポーネントが表示された時にAPIを呼び出す
    async function fetchShops() {
      try {
        setLoading(true)
        const data = await getShops()
        setShops(data)
      } catch (err) {
        setError('店舗データの取得に失敗しました')
        console.error(err)
      } finally {
        setLoading(false)
      }
    }
    
    fetchShops()
  }, [])  // 空配列 = 初回のみ実行

  // ローディング中はスピナーを表示
  if (loading) {
    return <div>読み込み中...</div>
  }

  // エラー時はエラーメッセージを表示
  if (error) {
    return <div style={{ color: 'red' }}>{error}</div>
  }

  // 通常のマップ表示
  return (
    // ... 既存のコード ...
  )
}
```

**初心者向けヒント**:

- `useEffect` は「コンポーネントが表示された時」に実行される
- `async/await` は「非同期処理」を書くための構文
- `try/catch` は「エラーが起きた時の処理」を書く

# 🔧 BEチーム（バックエンド）向けガイド

## BEチームとは？

バックエンド（BE）は、データを管理して、フロントエンドに提供する「サーバー側」を作るチームです。レストランの「厨房」のような役割です。

## 必要な知識・スキル

### 必須スキル

- **Node.js基礎**: JavaScriptをサーバーで動かす
- **Express基礎**: Webサーバーを作るフレームワーク
- **データベース基礎**: SQLの基本（SELECT, INSERT, UPDATE, DELETE）
- **REST API基礎**: HTTPメソッド（GET, POST, PUT, DELETE）の理解

### 学習リソース（初心者向け）

- [Node.js公式ドキュメント](https://nodejs.org/ja/docs) - 基礎から学べる
- [Express公式ガイド](https://expressjs.com/ja/starter/installing.html) - サーバーの作り方
- [Prisma入門](https://www.prisma.io/docs/getting-started) - データベース操作を簡単に

## 用語解説

- **API（Application Programming Interface）**: フロントエンドがデータを取得するための「窓口」
- **REST API**: Webでよく使われるAPIの設計方法
- **エンドポイント**: APIのURL（例: `/api/shops`）
- **データベース**: データを保存する場所（Excelのようなもの）
- **CRUD**: Create（作成）、Read（読み取り）、Update（更新）、Delete（削除）の4つの操作
- **マイグレーション**: データベースの構造を変更すること
- **ORM（Object-Relational Mapping）**: データベース操作を簡単にするツール

### タスク1: データベーススキーマ設計

**何をするか**: データベースにどんなテーブル（表）を作るかを決める**テーブル1: shops（店舗）**| カラム名 | 型 | 説明 | 例 ||---------|-----|------|-----|| id | INTEGER | 店舗ID（自動採番） | 1 || name | TEXT | 店舗名 | "海王焼きそば" || description | TEXT | 説明文 | "伝統の味！" || latitude | REAL | 緯度 | 35.661234 || longitude | REAL | 経度 | 139.791234 || image_url | TEXT | 画像のURL | "/images/yakisoba.jpg" || created_at | DATETIME | 作成日時 | "2024-01-01 10:00:00" || updated_at | DATETIME | 更新日時 | "2024-01-01 10:00:00" |**テーブル2: events（イベント）**| カラム名 | 型 | 説明 | 例 ||---------|-----|------|-----|| id | INTEGER | イベントID（自動採番） | 1 || time | TEXT | 時間 | "10:00" || title | TEXT | タイトル | "開会式" || location | TEXT | 場所 | "メインステージ" || description | TEXT | 説明文 | "海王祭の開会を祝う式典" || is_now | BOOLEAN | 開催中かどうか | true || created_at | DATETIME | 作成日時 | "2024-01-01 10:00:00" || updated_at | DATETIME | 更新日時 | "2024-01-01 10:00:00" |**初心者向けヒント**:

### タスク2: APIエンドポイント設計

**何をするか**: どんなURLで、どんなデータを返すかを決める**店舗関連のエンドポイント**:| メソッド | URL | 説明 | レスポンス例 ||---------|-----|------|------------|| GET | `/api/shops` | 店舗一覧を取得 | `[{id: 1, name: "..."}, ...]` || GET | `/api/shops/:id` | 店舗詳細を取得 | `{id: 1, name: "...", ...}` || POST | `/api/shops` | 店舗を新規作成 | `{id: 2, name: "...", ...}` || PUT | `/api/shops/:id` | 店舗を更新 | `{id: 1, name: "更新後", ...}` || DELETE | `/api/shops/:id` | 店舗を削除 | `{message: "削除しました"}` |**イベント関連のエンドポイント**: 同様のパターン**初心者向けヒント**:

- GET = データを取得（読む）
- POST = データを新規作成（書く）
- PUT = データを更新（書き換える）
- DELETE = データを削除（消す）

# 🗺️ アセットチーム（地図画像）向けガイド

## アセットチームとは？

アセットチームは、アプリで使う「画像」や「データ」を作るチームです。特に「地図画像」と「座標マッピング」を担当します。

## 必要な知識・スキル

### 必須スキル

- **画像編集**: Photoshop / GIMP / Figma の基本操作
- **座標の理解**: 緯度・経度の基本（Google Mapsで確認できる）
- **JSON形式**: データを書く形式（テキストエディタで編集可能）

### 学習リソース（初心者向け）

- [Google Mapsで座標を確認する方法](https://support.google.com/maps/answer/18539) - 緯度経度の取得方法
- [JSON形式の基本](https://developer.mozilla.org/ja/docs/Learn/JavaScript/Objects/JSON) - データの書き方

## 用語解説

- **解像度**: 画像のサイズ（ピクセル数）。大きいほど細かく表示できる
- **WebP**: 画像のファイル形式。ファイルサイズが小さい
- **座標マッピング**: 画像上の位置（ピクセル）と実際の位置（緯度経度）を対応付けること
- **タイル化**: 大きな画像を小さな画像に分割すること（必要に応じて）

**基本解像度**: 2048x2048px 以上（推奨: 4096x4096px）**理由**:

- 大きすぎると読み込みが遅くなる
- 小さすぎると拡大した時にぼやける

**高解像度版**: 8192x8192px（必要に応じて）**初心者向けヒント**:

- まずは4096x4096pxで作成
- 後から調整可能

## リスクと対策

### リスク1: 座標マッピングの精度

- **対策**: アセットチームは第2週に座標測定を完了し、第3週にFEチームと共同で精度確認

### リスク2: API連携の遅延

- **対策**: FEチームは第2週にモックサーバー（JSON Server等）を使用して先行開発可能

### リスク3: 地図画像のサイズが大きすぎる

- **対策**: アセットチームは画像最適化を実施（WebP、圧縮）。必要に応じてタイル化を検討

### リスク4: 統合時の不具合

- **対策**: 第3週に十分な統合テスト時間を確保。毎日の進捗共有会議を実施
